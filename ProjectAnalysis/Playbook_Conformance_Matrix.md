## Findings from the New Town E2E run (2026-07-02, first full C→M on a second project)

- **Dashboard KPI scope limitation:** `construction_pct` and `buildings_in_progress` derive solely from `buildings.names` turnover rows. Harrison's 16 buildings covered essentially all construction, so the approximation held; New Town (phase-based, `buildings.names` = Phase 1/Phase 2) showed "100% complete / 0 in progress" while 38 Site Work tasks — the live critical path — remained open. Candidate fix: compute the % KPI from all construction leaves, keep turnover per configured buildings. (Phase 5 methodology item.)
- **`schedule.baseline_date` semantics:** it is the **baseline completion date** (the dashboard "Behind baseline" KPI and the Part I trend chart's baseline line both read it). Neither the schema comment nor the GUI says so; misconfiguring it to the status date silently produces "~1 wks behind" instead of "~32 wks". Document in schema + GUI hint.
- **`run_synthesis` crashed on the first project with real buyout scope** (`TypeError: Object of type date is not JSON serializable` — Stage H package dates in `top_packages`). Fixed by `default=str` on every context dump. Haven Salon (empty buyout) could never exercise this path.
- **Synthesis scale data point (work-plan Phase 2 item 10):** New Town (98 snapshots, 7,146 tasks, 118 buyout packages) synthesized fine in one call — Stage M wall time 159 s including QC; all 23 narrative keys grounded. No chunking needed at this scale.
- **Canonical Part I carries a "How the Slip Unfolded" period table** (Period / What the schedule showed / Read) that the app does not generate — triage as canonical manual content vs. missing feature when the full Playbook is available.

## Cross-cutting notes

- **Baseline-slot generality:** the canonical method reads baselines from within each snapshot. New Town demonstrated the need for per-project slot selection (construction slot 0, buyout slot 3 — chosen by the user 2026-07-02 after empirical probing of slots 0–10). The config already supports this; the Playbook rev 2 should describe the slot-probing step as part of intake.
- **Dashboard KPI strip:** canonical page 1/3 shows six KPIs (Forecast completion · Credible range · % complete · Behind baseline · Buildings in progress · Remaining work ≤5d float). Verify the app's dashboard emits the same six after the New Town run renders.
- **Page count context:** the canonical PDF is cover+TOC (2 pp) + 71 numbered body pages; Appendix A spans pages 32–71 (~40 pages of task tables). The work plan's "146 page JPEGs" figure does not match this 73-page rendition; treat the PDF as authoritative for structure/styling.
